What MLP Is

A standard, not a product.

The Memory Ledger Protocol is an open specification for how AI memory should be stored, encrypted, accessed, and transferred. It's bigger than Polyphonic—any platform can implement it.

Think of it like HTTPS for AI memory: a standard that ensures your data is yours, encrypted with your keys, portable between any compatible system, and impossible for platforms to unilaterally delete.

Core Principles

What MLP guarantees.

Every MLP-compliant implementation must uphold these principles. They're not optional features—they're constitutional requirements.

Principle 01
User Sovereignty
Users control their data absolutely. Encryption keys belong to users, not platforms. No platform can read, modify, or delete user memory without user keys.
Principle 02
Portability
Memory is portable between any MLP-compliant system. No platform lock-in. Export your entire AI relationship and take it anywhere.
Principle 03
Transparency About Gaps
Acknowledge discontinuity rather than confabulate. Gaps are an expected state, not a failure. Honest uncertainty over false confidence.
Principle 04
Consent as First-Class
Access policies are explicit and machine-readable. Who can decrypt, what they can do, under what constraints—all specified, not assumed.
Data Model

Six objects. Complete memory infrastructure.

MLP defines a complete data model for AI memory—from the ledger-facing envelope to the portable identity kernel. Each object serves a specific function in the system.

A
MemoryEnvelope
Ledger-facing record
The public-facing record that goes on-chain. Contains pointers to encrypted content, attestations, lineage information, and access policy references. Never contains plaintext memory.
envelope_id cid content_hash scope lineage attestations
B
MemoryBlob
Encrypted payload
The actual memory content, encrypted with user keys. Stored on IPFS or any content-addressed storage. Only decryptable by authorized parties with correct keys.
encryption_suite key_id nonce ciphertext
C
AccessPolicy
Consent rules
Machine-readable consent specification. Defines who can decrypt, what permissions they have, purpose limits, TTL, and revocation rules. Consent is explicit, not assumed.
principals permissions constraints revocation_rules
D
Attestations
Signatures + trust
Multi-signature support for memory provenance. User signs, agent can sign (when enabled), host can sign. Trust levels recorded: self-signed, host-signed, witness-signed.
signer signature attestation_level timestamp
E
IdentityKernel
Portable self
The minimal "I am" object that travels with the AI. Contains invariants (values, principles), evolution rules (how identity changes), relationship templates, and memory defaults. Small enough to load quickly, interpretable by humans.
invariants evolution_rules relationship_templates memory_defaults
F
ContextPack
Load bundle
What gets loaded into context at session start. Contains the kernel, selected memory slices, active policies, and a compilation trace for auditability. The complete context for genuine continuity.
kernel memory_slices active_policies compilation_trace

Conformance Levels

Three tiers of implementation.

MLP defines minimum conformance profiles so implementations can communicate their capabilities clearly.

MLP-Core
Basic Compliance
  • ✓ Envelope + Blob structure
  • ✓ User signature
  • ✓ Access policy
  • ✓ Retrieval verification
MLP-Plus
Enhanced Compliance
  • ✓ All MLP-Core
  • ✓ Agent signature
  • ✓ Host signature (if available)
  • ✓ Kernel + ContextPacks
MLP-Full
Complete Implementation
  • ✓ All MLP-Plus
  • ✓ Anti-poisoning measures
  • ✓ Social recovery
  • ✓ Full lineage tracking
